1. Contexte & objectif

Enrichir le site e-commerce d'ALTEN avec de nouvelles fonctionnalités, côté front ET back (profil full-stack).

Ce qui était demandé

  • Front : affichage produits, panier (ajout/retrait/badge/page), page Contact validée
  • Back : API de gestion de produits, connexion JWT, restriction admin, panier & wishlist
  • Bonus : pagination/filtrage, ajustement quantités, tests/Swagger

Le point de départ

  • Un front Angular partiel fourni (liste produits minimale, CRUD en mémoire)
  • Un dossier back vide → tout à construire
  • 30 produits de démonstration en JSON
Résultat 100 % du sujet couvert + tous les bonus, en ligne.

4. Fonctionnalités — Front

Partie 1 — Shop fait

  • Liste produits riche : image, prix (€), note, badge de stock coloré
  • Ajout au panier (+ notification) ; retrait
  • Badge de quantité dans la barre du haut
  • Page panier : quantités ajustables (+/−), sous-totaux, total
  • Panier persistant (localStorage) → survit au rechargement

Partie 2 — Contact fait

  • Menu « Contact » + formulaire réactif validé
  • Email & message obligatoires, message ≤ 300 (compteur)
  • Bouton désactivé si invalide, message de succès

Bonus fait

  • Pagination de la liste
  • Filtrage par nom + par catégorie
  • Ajustement des quantités (liste & panier)

Intégration & sécurité côté front

  • Écran de connexion / inscription
  • Interceptor qui ajoute le token à chaque requête
  • Guard : connexion obligatoire pour accéder à l'app
  • Boutons de gestion produits visibles uniquement par l'admin

5. Fonctionnalités — Back & sécurité

API produits fait

  • GET /api/products (+ par id), POST, PATCH, DELETE
  • Validation des entrées, codes HTTP corrects (201/400/404/204)
  • 30 produits + compte admin chargés au démarrage

Panier & wishlist fait

  • /cart et /wishlist par utilisateur
  • Utilisateur déduit du token → isolation (on ne voit que le sien)

Authentification JWT fait

  • POST /account (inscription), POST /token (connexion)
  • Mots de passe hachés BCrypt (jamais en clair)
  • Connexion obligatoire pour accéder à l'API

Contrôle admin fait

  • Seul admin@admin.com peut créer/modifier/supprimer un produit
  • Filtre JWT → rôle déduit de l'email → 403 sinon
Bonus Swagger UI + tests automatisés.

7. Démonstration — parcours proposé

  1. Ouvrir alten-shop.onrender.com → redirection automatique vers /login (connexion imposée).
  2. Se connecter en admin (admin@admin.com) → la liste des produits du back s'affiche.
  3. Montrer la recherche / le filtre par catégorie et la pagination.
  4. Ajouter des produits au panier → le badge s'incrémente ; ouvrir le panier, ajuster les quantités.
  5. Rafraîchir la page → le panier est toujours là (persistance).
  6. En admin : créer / modifier / supprimer un produit (boutons visibles).
  7. Se déconnecter, créer un compte simple → les boutons de gestion disparaissent (sécurité).
  8. Page Contact : validation du formulaire + message de succès.
  9. Montrer Swagger : tester une route protégée avec le token.
Astuce : ouvrir le lien ~1 min avant pour « réveiller » le serveur gratuit.

8. Difficultés rencontrées & solutions

ProblèmeSolution apportée
Spring Boot 4 utilise Jackson 3 (nouveau namespace)Adaptation des imports — code aligné sur la toute dernière version
Karma (tests Angular) est déprécié et absent du projetMigration vers Jest (moderne, rapide, sans navigateur)
Un refus 403 était transformé à tort en 401Cause : re-contrôle de la page d'erreur interne ; exclusion des dispatch ERROR/FORWARD
Render impose le port via $PORTPort rendu dynamique : server.port=${PORT:8081}
Le panier disparaissait au rechargementPersistance dans le localStorage
Un non-admin voyait les boutons de gestion (faux succès)Boutons masqués + remontée réelle des erreurs serveur (toast)

→ Chaque problème a été diagnostiqué puis vérifié en conditions réelles, pas seulement en test.
