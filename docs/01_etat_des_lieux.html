ALTEN SHOP — État des lieux du projet

Analyse de l'existant : code fourni, technologies, architecture et fonctionnement actuel.

Angular 18 PrimeNG 17 TypeScript 5.4 Signals Standalone components Back-end : vide

1. Contexte & objectif

Le dépôt est un test technique de recrutement ALTEN. La consigne (fichier README.md à la racine) décrit un mini site e-commerce, « ALTEN SHOP », pour lequel le candidat doit compléter un front-end Angular et/ou créer un back-end de gestion de produits.

Trois profils sont prévus dans la consigne :

  • Front-end : enrichir l'application Angular fournie (panier, page contact, etc.).
  • Back-end : développer une API de gestion de produits (techno au choix) + authentification.
  • Full-stack : les deux, avec en plus une stratégie de test de l'API.
À retenir pour l'entretien
Le projet livré est un squelette de départ : seul le front est amorcé (CRUD produit partiel), et le back-end n'existe pas encore. Le travail attendu consiste à compléter ce socle.

4. Architecture front-end

L'application suit un découpage par fonctionnalité (feature) avec une convention de couches qui rappelle les bonnes pratiques Angular modernes :

CoucheDossierResponsabilité
data-accessproducts/data-accessModèles (product.model.ts) et service de données (products.service.ts) — communication HTTP + état
featuresproducts/features, shared/featuresComposants « pages » intelligents, routés (product-list, home)
uiproducts/ui, shared/uiComposants présentationnels réutilisables (product-form, panel-menu)
sharedshared/Éléments transverses (accueil, menu de navigation)

Layout global

Défini dans app.component.html : une toolbar en haut (logo + titre « ALTEN SHOP » + icône panier statique), puis un splitter 20 % / 80 % : menu latéral à gauche (app-panel-menu), contenu routé à droite (router-outlet).

Routage & lazy loading

// app.routes.ts
home      → HomeComponent (chargement direct)
products  → loadChildren(...products.routes)   // chargé à la demande
""        → redirige vers home

// products.routes.ts
products/list → ProductListComponent
products/**   → redirige vers list

Le module produits est chargé paresseusement (loadChildren), bonne pratique pour le découpage du bundle. Le menu (panel-menu.component.ts) expose deux entrées : Accueil (/home) et Produits (/products/list).

5. Détail des composants existants

ComposantTypeCe qu'il fait
AppComponentracine, standaloneLayout (toolbar + splitter + menu + outlet). Titre « ALTEN SHOP ».
HomeComponentfeatureSimple carte « Bienvenue sur ALTEN SHOP ! ». Page d'accueil minimale.
PanelMenuComponentuiMenu latéral PrimeNG avec 2 liens (Accueil, Produits).
ProductListComponentfeaturePage liste produits + orchestration du CRUD (création, édition, suppression via dialog).
ProductFormComponentuiFormulaire d'ajout/édition (nom, prix, description, catégorie) dans un dialog.

ProductListComponent — le cœur fonctionnel actuel

  • Au ngOnInit, appelle productsService.get() pour charger la liste.
  • Affiche les produits via un p-dataView + p-card : seulement la catégorie et le nom sont affichés (prix, image, stock… non affichés).
  • Boutons Créer, Modifier, Supprimer ; le formulaire s'ouvre dans un p-dialog.
  • Gère un signal editedProduct et deux drapeaux (isDialogVisible, isCreation) pour distinguer création et édition.

ProductFormComponent — formulaire

  • Reçoit le produit via input.required<Product>() (nouvelle API de signaux d'input).
  • Émet save et cancel via @Output.
  • Travaille sur une copie : editedProduct = computed(() => ({...product()})) — évite de muter directement l'objet parent.
  • Champs gérés : nom (requis), prix (requis), description, catégorie (dropdown figé : Accessories / Fitness / Clothing / Electronics).
Point d'attention technique
Le formulaire ne couvre que 4 champs sur les 14 du modèle Product (pas de code, image, quantity, inventoryStatus, rating, etc.). À compléter selon les besoins du back-end.

6. Gestion de l'état — Signals + Service

Le ProductsService (fourni providedIn:'root', donc singleton) centralise l'état avec un signal privé exposé en lecture seule :

private readonly _products = signal<Product[]>([]);
public  readonly products  = this._products.asReadonly();

Quatre méthodes HTTP, chacune mettant à jour le signal local après l'appel :

MéthodeAppel HTTPComportement
get()GET /api/productsSi échec → fallback sur assets/products.json. Remplit le signal.
create()POST /api/productsSi échec → renvoie true. Ajoute le produit en tête de liste.
update()PATCH /api/products/:idSi échec → renvoie true. Remplace le produit dans la liste.
delete()DELETE /api/products/:idSi échec → renvoie true. Retire le produit de la liste.
Piège à connaître (question probable du recruteur)
Le catchError de create/update/delete masque les erreurs réseau et renvoie toujours « succès », puis met quand même à jour l'état local (mise à jour « optimiste »). C'est pratique pour démonter le front sans back-end (tout marche en mémoire), mais en production cela ferait croire à l'utilisateur qu'un enregistrement a réussi alors qu'il a échoué. À corriger une fois le vrai back branché.

7. Modèle de données

interface Product {
  id: number;
  code: string;
  name: string;
  description: string;
  image: string;
  category: string;
  price: number;
  quantity: number;
  internalReference: string;
  shellId: number;
  inventoryStatus:
    "INSTOCK"|"LOWSTOCK"|"OUTOFSTOCK";
  rating: number;
  createdAt: number;
  updatedAt: number;
}

Le modèle Product (côté front) est identique à celui imposé par la consigne back-end du README. C'est le contrat partagé entre front et futur back.

Les données de démo (assets/products.json) contiennent 30 produits (montres, vêtements, accessoires…) avec ces champs renseignés, ids à partir de 1000.

Note : quantity existe dans le modèle mais n'est pas affiché ni édité actuellement — utile pour le futur panier et la gestion de stock.

9. Le back-end

État : inexistant
Le dossier back/ ne contient qu'un fichier .gitkeep (placeholder Git). Aucune ligne de code serveur. Tout est à créer : techno, API, base de données, authentification.

Conséquence : aujourd'hui, l'application fonctionne uniquement en mode mock. Les appels /api/... échouent et le service bascule sur le JSON local. La persistance n'est donc pas réelle (rechargement de page = retour aux 30 produits d'origine).

11. Points d'attention à savoir défendre en entretien

SujetCe qu'il faut pouvoir dire
Pourquoi des Signals ?État réactif natif Angular 16+, plus simple que NgRx pour ce périmètre, granularité fine, pas de boilerplate. Le service expose un état en lecture seule (asReadonly) → encapsulation.
Standalone componentsPlus de NgModules : chaque composant déclare ses propres imports. Recommandé depuis Angular 17, simplifie l'arbre de dépendances et le lazy-loading.
Le fallback JSONPermet de développer le front sans attendre le back. Mais le catchError sur les mutations masque les vraies erreurs → à durcir en prod (remonter l'erreur, toast d'échec).
Lazy loadingloadChildren sur la feature produits → le code n'est téléchargé qu'à la navigation. Bon pour le temps de chargement initial.
PrimeNG vs maisonGain de temps (composants riches accessibles prêts à l'emploi : DataView, Dialog, Dropdown). Cohérent avec un contexte ESN/livraison rapide.
BiomeOutil unique (lint+format) écrit en Rust, très rapide ; remplace ESLint+Prettier. Montre une stack outillage moderne.
